# **ТЕХНИЧЕСКИЙ ПРОЕКТ: TaskGraph (EduFlow)**

## **Семантический граф-ориентированный таск-трекер с ИИ-генератором зависимостей задач**

## **1\. Введение и бизнес-логика (Суть проекта)**

### **1.1. Проблема**

Классические методологии управления проектами (Agile/Scrum) и их инструменты (Jira, Trello, Kaiten) ориентированы на опытных разработчиков и плоские структуры задач. При обучении или создании первого пет-проекта студенты и начинающие разработчики сталкиваются со следующими проблемами:

* **Паралич планирования:** Сложно самостоятельно разбить абстрактную цель (например, «Настроить авторизацию на Spring Security») на последовательность конкретных, атомарных шагов.  
* **Нарушение технологической последовательности:** Попытки реализовать зависимые фичи (например, логику корзины на фронтенде) до готовности фундамента (API эндпоинтов бэкенда), что приводит к хаосу в коде и постоянному рефакторингу.  
* **Информационный шум:** Поиск качественной документации по каждой технологии занимает много времени, а общие ИИ-чаты дают абстрактные советы без привязки к конкретному шагу разработки.

### **1.2. Решение: TaskGraph**

TaskGraph переосмысляет концепцию таск-трекера. Вместо списков и канбан-досок проект представляется в виде **направленного ациклического графа задач (Directed Acyclic Graph — DAG)**.

Каждая задача — это узел графа. Связи (ребра) определяют жесткие технологические зависимости (задача *B* заблокирована, пока не выполнена задача *A*).

Интегрированный ИИ-агент выступает архитектором-методистом: он декомпозирует глобальную цель проекта в граф задач и снабжает каждый узел проверенной контекстной информацией.

TaskGraph выступает в роли интеллектуальной системы управления проектами и знаниями. Она не только визуализирует зависимости, но и обеспечивает прозрачность командной работы через механизмы распределения ресурсов, версионируемую Wiki-документацию и детальный аудит действий участников (как людей, так и ИИ-агентов). 

## **2\. Сценарии использования (Use Cases)**

### **Сценарий 1: Создание проекта и ИИ-декомпозиция**

1. **Пользователь** авторизуется в системе и нажимает кнопку «Создать проект».  
2. Система предлагает форму ввода:  
   * *Название проекта* (например, E-Commerce Store).  
   * *Стек бэкенда* (например, Java, Spring Boot).  
   * *Стек фронтенда* (например, TypeScript, React, Redux Toolkit).  
   * *Описание цели* (например, «Создать MVP интернет-магазина с авторизацией, каталогом товаров, корзиной и оплатой»).  
3. **ИИ-Агент** обрабатывает запрос и генерирует древовидный граф задач с логическими связями.  
4. **Пользователь** видит на экране интерактивный граф, где активны только корневые задачи (развертывание окружения, инициализация репозиториев), а остальные узлы заблокированы (помечены серым цветом).

### **Сценарий 2: Продвижение по графу и изменение состояний**

1. **Пользователь** выбирает доступную (разблокированную) задачу Инициализация проекта.  
2. В боковой панели он изучает сгенерированные ИИ чек-листы и ссылки на документацию.  
3. Пользователь переводит задачу в статус В работе, а по завершении — в статус Выполнено.  
4. **Бэкенд-движок графа** проверяет зависимости. Так как задача Инициализация проекта завершена, система автоматически разблокирует дочерний узел и уведомляет пользователя.

### **Сценарий 3: Динамическая мутация графа (Форс-мажор)**

1. В середине проекта **Пользователь** решает добавить новую технологию (например, «Добавить кэширование через Redis для каталога»).  
2. Пользователь открывает ИИ-ассистента проекта и пишет запрос: *«Хочу добавить Redis для кэширования продуктов»*.  
3. **ИИ-Агент** анализирует существующий граф проекта, находит оптимальное место для интеграции и предлагает изменения.  
4. После подтверждения пользователем, бэкенд встраивает новые узлы (Настройка Redis, Интеграция кэширования в сервисы) в существующий DAG, связывая их с базовой задачей по каталогу продуктов и блокируя зависимые задачи до их выполнения.

### 

### **Сценарий 4: Генерация и визуализация архитектурных диаграмм**

1. **Пользователь** завершает планирование проекта или хочет посмотреть на итоговую архитектуру системы.  
2. В интерфейсе он нажимает на вкладку **«Архитектурный рентген» (Architecture Blueprint)**.  
3. Система отправляет запрос на ИИ-мост, который анализирует структуру графа и сгенерированные зависимости.  
4. **ИИ-Агент** генерирует текстовое представление UML/C4 диаграмм в формате разметки **Mermaid.js**.  
5. **Пользователь** видит на интерактивном холсте вкладки с автогенерируемыми схемами:  
   * **C4 Diagram (System Context / Containers):** Показывает, как фронтенд взаимодействует с бэкендом, базой данных и внешними сервисами (например, платежным шлюзом).  
   * **Sequence Diagram (Диаграмма последовательностей):** Визуализирует сквозной жизненный цикл ключевых запросов (например, процесс авторизации пользователя: Клиент \-\> Spring Security Filter Chain \-\> База данных \-\> Генерация JWT \-\> Клиент).  
   * **Class/Component Diagram (Диаграмма классов/компонентов):** Отражает архитектуру классов бэкенда (контроллеры, сервисы, репозитории, DTO) и структуру связей компонентов на фронтенде.

### **Сценарий 5: Командное планирование и учет ресурсов**

6. **Владелец проекта** приглашает участников в рабочее пространство.  
7. При создании или мутации графа **ИИ-Агент** автоматически рассчитывает прогнозные трудозатраты (`estimatedHours`) для каждого узла (если эта функция включена, иначе заполняют сами участники).  
8. Участники назначают себя исполнителями на доступные (`AVAILABLE`) задачи. Аватары исполнителей отображаются непосредственно на узлах графа в ReactFlow.  
9. По мере выполнения пользователи логируют фактически затраченное время, что позволяет системе сравнивать план с фактом и визуализировать общий прогресс проекта.

### **Сценарий 6: Формирование базы знаний и аудит**

10. Для каждой крупной задачи автоматически создается связанная страница в **Project Wiki**.  
11. Участники наполняют Wiki техническими деталями, а ИИ-агент подгружает туда актуальные ссылки на документацию.  
12. Любое критическое изменение (смена статуса, перенос ребра графа, правка Wiki) фиксируется в **Action Log**, формируя прозрачную историю развития продукта, доступную для ретроспективы.

## **3\. Архитектура MVP (Микросервисный подход с ИИ-мостом)**

Для обеспечения гибкости, простоты работы с ИИ-моделями и масштабируемости MVP-архитектура разделяется на четыре четких уровня. Ключевым решением здесь является вынос всей логики оркестрации нейросетей в отдельный микросервис-мост (AI Bridge).

### **3.1. Клиентский уровень (Frontend Layer)**

Интерактивное SPA-приложение (Single Page Application) на React \+ TypeScript:

* **Интерфейс (ReactFlow):** Отрисовывает динамический граф, позиционирует узлы на основе расчетов бэкенда, отслеживает клики и смену статусов задач.  
* **Генератор диаграмм (Mermaid.js):** Легковесная библиотека, которая принимает текстовую разметку от ИИ-моста и на лету отрисовывает интерактивные векторные SVG-диаграммы (C4, Sequence, Class Diagrams) с поддержкой масштабирования.  
* **Панель настроек (Settings Panel):** Позволяет пользователю выбрать режим подключения ИИ — личный API-ключ Google Gemini (Bring Your Own Key) или локальный инстанс Ollama.

### **3.2. Основной серверный уровень (Core Backend Layer — Spring Boot)**

Основной бэкенд на Java, который полностью изолирован от деталей интеграции с конкретными LLM-библиотеками:

* **Обязанности:** Управление жизненным циклом проектов, хранение структуры графов в оперативной памяти (In-Memory), обработка бизнес-логики (смена статусов задач, вычисление разблокировок), авторизация пользователей и распределение ролей между участниками проекта,  API для управления Wiki-страницами и привязки их к метаданным узлов, система регистрации событий, формирующая хронологическую ленту изменений (User actions \+ AI modifications), агрегация трудозатрат по веткам графа и расчет процента завершенности проекта.  
* **Взаимодействие:** При необходимости декомпозиции проекта или генерации архитектурных диаграмм Spring Boot отправляет чистый REST-запрос на FastAPI-мост и получает строго валидированный JSON-ответ.

### **3.3. ИИ-Сервисный мост (AI Service Bridge — FastAPI)**

Легковесный сервис на Python (FastAPI), выступающий в роли «умного переходника» между бэкендом и миром ИИ:

* **Почему именно Python/FastAPI, а не прямой вызов из Java?**  
  1. *Экосистема ИИ:* Python является стандартом для работы с LLM. Библиотеки для работы с промптами, RAG, агентами (LangChain, Instructor, официальные SDK Gemini) обновляются на Python в первую очередь. Писать валидацию схем на Pydantic в разы проще, чем парсить JSON-структуры в Java.  
  2. *Гибкость:* FastAPI может переключать модели «на лету» (с облачного Gemini API на локальный Ollama), используя единый унифицированный интерфейс.  
  3. *Асинхронность:* Идеально подходит для фонового обогащения карточек задач (асинхронные неблокирующие вызовы к LLM).

### **3.4. Внешний интеллектуальный уровень (AI Providers)**

Слой ИИ-моделей, которые могут быть подключены к FastAPI-мосту в зависимости от конфигурации:

* **SaaS-решение (Google Gemini API через личные ключи):** Облачный провайдер, обеспечивающий высокую скорость генерации и строгое соответствие JSON-схеме, функционирующий по принципу BYOK (Bring Your Own Key).  
* **Локальное решение (Ollama / Local LLM):** Локально запущенная пользователем модель (например, Llama 3, Mistral или Qwen), доступная по локальной сети без отправки данных на внешние сервера.

### **3.5. Схема сквозного взаимодействия (Data Flow)**

1. **Инициация:** Пользователь отправляет описание проекта через React-интерфейс, прикрепив личный API-токен или выбрав локальный режим.  
2. **Запрос к ИИ:** Бэкенд на Spring Boot принимает запрос, оборачивает его в системные инструкции с требованиями актуальности версий и пересылает API-токен пользователя в HTTP-заголовках в FastAPI-мост, который запрашивает скелет проекта у выбранного ИИ-провайдера.  
3. **Парсинг и сохранение:** Бэкенд получает структурированный JSON, валидирует его на ацикличность, генерирует уникальные идентификаторы для задач, сохраняет структуру проекта в память и производит топологическую сортировку для расчета визуальной сетки.  
4. **Рендеринг:** React-клиент получает готовую структуру, позиционирует узлы на холсте ReactFlow и активирует доступные для работы стартовые задачи.  
5. **Отрисовка Архитектуры (по запросу):** При переходе во вкладку диаграмм FastAPI запрашивает у ИИ код разметки Mermaid на основе текущих задач проекта, возвращает его на фронтенд, где библиотека Mermaid.js мгновенно визуализирует схемы C4 и UML последовательностей.

## **4\. Ключевые алгоритмы и логика графа**

### **4.1. Жизненный цикл задачи и авторазблокировка**

Статус узлов динамически пересчитывается при изменении состояния связанных задач:

* **Инициализация:** Все задачи без входящих связей (зависимостей) получают статус AVAILABLE. Все остальные — LOCKED.  
* **Trigger разблокировки:** Как только задача переходит в статус COMPLETED, бэкенд находит все её исходящие связи. Для каждого дочернего узла проверяется наличие других незавершенных «родителей». Если все предшествующие задачи выполнены, дочерняя задача автоматически разблокируется (AVAILABLE).

### **4.2. Контроль целостности (Поиск циклов)**

При ручном добавлении новых связей между задачами или при их изменении пользователем бэкенд проверяет граф на отсутствие **циклических зависимостей**.

* **Реализация:** Алгоритм обхода в глубину (DFS) с цветовыми метками узлов (белый/серый/черный). Если в процессе обхода по цепочке зависимостей алгоритм натыкается на узел, который уже находится в процессе исследования (серый), фиксируется наличие цикла, и бэкенд отклоняет сохранение связи.

Система реализует механизм **Smart Recovery** для автоматического разрешения конфликтов. Если предложенная мутация или ручная правка создает циклическую зависимость, Core API не просто блокирует действие, а инициирует повторный запрос к AI Bridge. В этот запрос передается текущая структура и описание конфликта. ИИ-агент анализирует петлю и предлагает альтернативный вариант встраивания задачи (например, декомпозицию конфликтного узла на два под-этапа), что позволяет сохранить ацикличность графа (DAG) без участия пользователя 

### **4.3. Топологическая сортировка и уровни визуализации**

Чтобы граф на фронтенде выглядел упорядоченно, бэкенд выполняет топологическую сортировку графа (например, алгоритм Кана). Это позволяет рассчитать "уровень глубины" (слой) для каждой задачи и передать на фронтенд готовые координаты для автоматического позиционирования узлов в ReactFlow.

## **5\. Интеграция с ИИ: Токены, FastAPI и локальные модели**

Разделение архитектуры на Spring Boot и FastAPI позволяет гибко настроить безопасность и поддержку приватных локальных моделей силами самого пользователя.

### **5.1. Стратегии подключения моделей и авторизации**

В рамках Open-Source концепции система поддерживает две полностью автономные и безопасные стратегии подключения моделей:

1. **Личный ключ пользователя (Bring Your Own Key — BYOK):**  
   * *Как работает:* Пользователь указывает собственный API-ключ Google Gemini в панели настроек веб\-интерфейса (React). Этот ключ не сохраняется в базе данных бэкенда. При отправке запросов на генерацию или декомпозицию ключ передается в HTTP-заголовках (X-Gemini-API-Key) через Spring Boot в FastAPI-мост. FastAPI совершает авторизованный запрос к облачным серверам Google от лица пользователя.  
2. **Локальная модель (Local LLM via Ollama):**  
   * *Как работает:* Пользователь разворачивает у себя на локальной машине или в домашней сети утилиту **Ollama** с открытой LLM (например, llama3, mistral или qwen). В настройках интерфейса выбирается режим "Local Ollama" и указывается хост-адрес (по умолчанию http://localhost:11434). FastAPI-мост напрямую взаимодействует с локальной моделью по её стандартному API без отправки конфиденциального кода или идей проекта во внешнюю сеть.

### **5.2. Обеспечение актуальности данных (Делегированная модель)**

Проект функционирует на принципах разделения зон ответственности. Система не навязывает жестких алгоритмов поиска в реальном времени, а перекладывает эту задачу на возможности выбранной пользователем модели:

* **При работе с облачным API (например, Google Gemini):** Система принудительно включает флаг веб\-поиска (Google Search Grounding) в конфигурации запроса к API. Модель перед генерацией графа самостоятельно запрашивает актуальные версии библиотек и документацию в реальном времени.  
* **При работе с локальной моделью (например, Ollama):** Актуальность решений напрямую зависит от версии и качества весов выбранной пользователем модели (рекомендуется использовать наиболее свежие версии моделей, запущенные локально). Системные промпты в ИИ-мосте снабжены строгими инструкциями по использованию актуального синтаксиса популярных технологий.

### **5.3. Двухфазный конвейер генерации графа (Prompt Chaining)**

Генерация всего графа с глубоким описанием за один проход снижает качество и ведет к «лени» модели. Процесс разделен на шаги:

* **Фаза 1: Скелет графа (Быстрый ответ)**  
  Бэкенд просит Gemini/Ollama сгенерировать *только структуру графа* (список узлов с их названиями и связи между ними) строго по JSON Schema. Этот запрос выполняется за 2-3 секунды, и пользователь мгновенно видит интерактивную карту проекта в браузере.  
* **Фаза 2: Асинхронное фоновое обогащение (Глубокий контекст)**  
  После сохранения скелета бэкенд в фоновом режиме отправляет точечные асинхронные запросы к ИИ для детализации *каждой задачи отдельно*. ИИ наполняет карточки детальными чек-листы, подводными камнями и ссылками на документацию.  
* **Фаза 3: Динамический патчинг (Мутация)**  
  При запросе на добавление новой функциональности ИИ отправляется текущая структура (без детального контента), а ИИ возвращает только наборы новых узлов и связей, которые бэкенд интеграционно встраивает в существующий граф без полной перегенерации проекта.

### **5.4. Контракт передачи данных для диаграмм**

Для генерации архитектурных диаграмм FastAPI просит Gemini выдать разметку в текстовом формате Mermaid.js, соответствующую UML-стандартам:

* c4Code — текстовый блок для C4 Context Diagram.  
* sequenceCode — текстовый блок для Sequence Diagram.  
* classCode — текстовый блок для Class Diagram.

Фронтенд считывает эти блоки и скармливает их компоненту \<Mermaid /\> для отрисовки интерактивной графики.

### **5.5. Генерация метаданных и трудозатрат** 

В процессе декомпозиции (Фаза 1 и 2\) ИИ-агент выполняет роль технического лида. Он не только определяет структуру, но и: 

* Оценивает сложность каждого узла в часах, исходя из объема необходимых работ.  
* Определяет тип задачи (Backend, Frontend, Devops, Documentation), что позволяет системе автоматически предлагать подходящих исполнителей из команды.

## **6\. Этапы реализации проекта**

### **Фаза 1: Интерактивный граф (MVP)**

* **Результат:** Пользователь может вручную создавать проект, добавлять задачи, визуально соединять их стрелочками на холсте ReactFlow. Бэкенд сохраняет эту структуру в памяти, проверяет отсутствие циклов и автоматически разблокирует дочерние задачи при завершении родительских.

### **Фаза 2: ИИ-декомпозитор проектов (FastAPI \+ Gemini/Ollama)**

* **Результат:** Запуск микросервиса FastAPI. Реализация двухфазного конвейера генерации. Интеграция Spring Boot с FastAPI. Возможность генерировать графы как через облако (с использованием личного API-ключа), так и локально через Ollama.

### **Фаза 3: Командная работа и контроль**

* **Результат:** Реализация системы ролей, назначение исполнителей на задачи, внедрение модуля Time-tracking (логирование часов) и Action Log (лента событий).

### **Фаза 4: Интеллектуальная Wiki и Knowledge Base** 

* **Результат:** Интеграция текстового редактора для Wiki-страниц, автоматическое связывание контента ИИ-подсказок с базой знаний проекта, экспорт всей накопленной документации в Markdown/PDF.

### **Фаза 5: Архитектурный рентген (UML / C4 via Mermaid.js)**

* **Результат:** Реализация вкладки «Архитектурный рентген» в React-интерфейсе. Интеграция библиотеки Mermaid.js. Разработка промптов на FastAPI для синтеза C4-схем, UML-диаграмм классов и последовательностей на основе структуры проекта. Пользователь может наглядно увидеть и экспортировать визуальную архитектуру своей будущей системы.

### **Фаза 6: Интеграция с GitHub и расширенный функционал**

* **Результат:** Интеграция с вебхуками GitHub (автоматическое закрытие задач по ID в коммите, например git commit \-m "fix security \#task-12"), организация совместной работы над одним графом для команды студентов и контекстный ИИ-помощник внутри задач.
